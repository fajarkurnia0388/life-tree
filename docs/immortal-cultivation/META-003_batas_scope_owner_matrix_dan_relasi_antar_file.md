## 2. Fungsi Dokumen

Dokumen ini berfungsi sebagai **mekanisme pengendalian batas** untuk seluruh arsip riset cultivation. Fungsi operasionalnya meliputi:

1. menetapkan file utama untuk setiap konsep besar;
2. menentukan file mana yang hanya boleh menyentuh konsep secara terbatas;
3. mencegah definisi inti terulang tanpa kendali;
4. membedakan antara file definisional, file aplikatif, file komparatif, dan file market;
5. menyediakan protokol ketika muncul topik baru atau overlap baru.

Dokumen ini tidak menggantikan isi file-file substantif. Dokumen ini hanya menetapkan **siapa membahas apa, sejauh mana, dan dalam relasi seperti apa**.

---

## 3. Prinsip Batas Scope

### 3.1 Prinsip owner concept
Setiap konsep besar memiliki **satu file owner** yang memegang otoritas definisional utama.

### 3.2 Prinsip sentuh-terbatas
File non-owner boleh menyentuh konsep owner hanya bila konsep tersebut diperlukan untuk fungsi file, tetapi:
- definisi inti tidak diulang penuh;
- pembahasan dibatasi pada implikasi lokal file;
- rujukan eksplisit ke file owner wajib diberikan.

### 3.3 Prinsip domain-separation
Konsep yang sama dapat muncul di beberapa domain, tetapi dengan fokus berbeda:
- **FOUND/CORE** → definisi, ontologi, fondasi
- **SYS/PATH/WORLD/ENT** → aplikasi, institusi, praktik, objek
- **NARR** → fungsi cerita dan pengalaman pembacaan
- **COMP** → perbandingan model antarkarya/subgenre
- **MRKT** → performa, resepsi, dan industri
- **TOOL** → desain ulang dan penerapan kreatif

### 3.4 Prinsip prioritas epistemik
Apabila terjadi konflik isi antara file dari domain berbeda, prioritas interpretatif awal mengikuti urutan berikut:

1. **META**
2. **FOUNDATION**
3. **CORE**
4. **SYSTEM**
5. **PATH / WORLD / ENTITY**
6. **NARRATIVE**
7. **COMPARATIVE**
8. **MARKET**
9. **TOOLKIT**

Urutan ini tidak berarti domain yang lebih rendah tidak penting; urutan ini hanya dipakai untuk menentukan **otoritas definisional awal**.

## 4. Protokol Rujukan

### 4.1 Aturan umum
- File owner mendefinisikan konsep.
- File lain menjelaskan hanya aspek yang diperlukan oleh domainnya.
- File comparative tidak mengambil alih definisi file owner.
- File market tidak mengambil alih penilaian ontologis atau naratif.
- File toolkit tidak dipakai sebagai sumber definisional historis.

### 4.2 Formula rujukan yang disarankan
Untuk menjaga konsistensi, file dapat memakai pola seperti:
- **Untuk definisi inti, lihat:** `CORE-001`
- **Untuk aplikasi jalur, lihat:** `PATH-004`
- **Untuk implikasi dunia, lihat:** `WORLD-004`
- **Untuk perbandingan antarkarya, lihat:** `COMP-005`

## 8. Aturan Penambahan File Baru

Bila ditemukan kebutuhan file baru, prosedurnya adalah sebagai berikut:

1. topik baru dicatat terlebih dahulu di `META-001`;
2. hubungan topik baru dengan owner concept yang sudah ada dicek di `META-003`;
3. ditentukan apakah file baru adalah Hub, Satellite, atau Bridge;
4. ditentukan apakah file baru Stable, Living, Snapshot, atau Draft;
5. penambahan baru hanya dilakukan bila topik tidak dapat ditangani memadai oleh file eksisting tanpa merusak batas scope.

File baru tidak boleh dibuat langsung tanpa pencatatan administratif awal.

---

## 9. Aturan Eskalasi Konflik Scope

Jika dalam proses penulisan ditemukan bahwa satu file mulai melebar ke wilayah file lain, urutan penyelesaiannya adalah:

1. cek owner concept;
2. cek matriks overlap di dokumen ini;
3. pangkas isi file non-owner dan ubah menjadi rujukan;
4. bila konsep memang terlalu besar, usulkan pecahan file baru melalui `META-001` dan revisi `META-003`.

---

## 10. Format Ringkas Front-Matter yang Disarankan

Setiap file substantif disarankan membuka dokumen dengan blok minimum seperti:

- ID
- Domain
- Role/Status
- Pertanyaan inti
- Owner concept
- Does not cover
- Related files
- Last updated

Front-matter ini berfungsi sebagai alat anti-overlap.
